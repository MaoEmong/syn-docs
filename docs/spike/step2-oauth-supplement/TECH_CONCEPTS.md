## 1. Jackson 3 (데이터 변환의 핵심)
**Jackson**은 자바 객체와 JSON 데이터 사이를 이어주는 통역사 역할을 합니다.

### 왜 중요한가요?
우리 서버가 클라이언트와 통신할 때 모든 데이터는 JSON으로 오고 갑니다. 자바 코드로 데이터를 처리하려면 이 JSON을 자바 객체로 바꿔야 하는데, 그 일을 Jackson이 담당합니다.

### Jackson 2에서 3로의 변화 (중요!)
이번 프로젝트는 최신 버전인 **Jackson 3**를 사용합니다.
- **이름표의 변화**: 기존 `com.fasterxml.jackson` 패키지 이름이 `tools.jackson`으로 바뀝니다. 코드를 짤 때 자동완성을 주의해야 합니다.
- **불변성 강조**: 예전에는 `ObjectMapper`를 막 수정해서 썼지만, 이제는 `JsonMapper`라는 것을 만들 때(Builder 패턴) 모든 설정을 끝내고 변경할 수 없게 만들어 성능과 안전성을 높였습니다.
- **강력한 보안**: 모르는 데이터 타입을 함부로 변환하지 못하도록 하는 보안 기능이 기본으로 켜져 있어, 해커의 공격으로부터 더 안전합니다.

## 2. 멀티테넌시 (Tenant - 다중 고객 관리)
**멀티테넌시(Multi-tenancy)**는 하나의 아파트(서버)에 여러 가구(테넌트/고객사)가 독립적으로 사는 것과 같은 구조입니다.

### 테넌트(Tenant)란?
우리 서비스를 이용하는 한 단위의 고객사(예: A회사, B대학교)를 말합니다. 각 테넌트는 서로의 데이터가 절대 보이지 않아야 합니다.

### 구현 방식 비교
1.  **DB 분리 (Silo)**: 각 고객사마다 실제 데이터베이스를 따로 만듭니다. 가장 안전하지만 비용이 많이 듭니다.
2.  **스키마 분리 (Bridge)**: 한 DB 안에서 구역(Schema)만 나눕니다. 관리가 조금 더 편합니다.
3.  **컬럼 분리 (Pool)**: 모든 데이터를 한 테이블에 넣되, `tenant_id`라는 기둥을 세워 구분합니다. 우리 프로젝트처럼 많은 고객을 효율적으로 관리할 때 주로 씁니다.

## 3. Flyway (데이터베이스의 타임머신)
**Flyway**는 데이터베이스의 변경 사항을 소스코드처럼 버전별로 기록하고 관리하는 도구입니다.

### 왜 쓰나요?
A 개발자가 테이블에 '나이' 컬럼을 추가했는데, B 개발자는 이를 모르면 코드가 에러 납니다. Flyway는 `V1__이름변경.sql`, `V2__나이추가.sql` 처럼 번호표를 붙인 SQL 파일을 만들면, 서버가 켜질 때 자동으로 실행해서 모든 개발자의 DB 상태를 똑같이 맞춰줍니다.

### 이름 짓기 규칙
- `V1__설명.sql`: 버전이 있는 마이그레이션 (한 번만 실행됨)
- `R__설명.sql`: 반복 가능한 마이그레이션 (파일 내용이 바뀌면 다시 실행됨)
- **주의**: 중간에 언더바(`__`)가 반드시 **두 개**여야 Flyway가 인식합니다.

## 4. 3-케이스 트랜잭션 (분산 시스템의 생존 전략)
분산 환경(MSA 등)에서 통신 결과를 단순히 '성공'과 '실패'로만 나누면 데이터가 꼬일 수 있습니다. 그래서 **'알 수 없음(Timeout)'**이라는 제3의 케이스를 반드시 고려해야 합니다.

### 세 가지 케이스 (3-Case)
1.  **성공 (Success)**: "결제 완료!" 응답을 받음. 마음 놓고 다음 단계로 진행.
2.  **실패 (Failure)**: "잔액 부족!" 응답을 받음. 즉시 중단하고 사용자에게 알림.
3.  **알 수 없음 (Unknown/Timeout)**: 요청을 보냈는데 한참 뒤에 "연결이 끊겼습니다"라고 뜸. **이게 처리가 된 건지 안 된 건지 모르는 게 가장 위험합니다.**

### 어떻게 해결하나요?
- **멱등성(Idempotency)**: 같은 요청을 두 번 보내도 결과는 한 번만 처리되게 만듭니다. (예: "100원 결제"를 실수로 두 번 보내도 실제로는 한 번만 출금됨)
- **확인 후 처리**: 결과가 모호할 때는 "방금 보낸 거 처리됐어?"라고 다시 물어보는 로직을 추가합니다.
